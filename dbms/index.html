Database Management System

DBMS is a means to manage resources of database(s).

Type of management vary from one DBMS to another, but in general:

  • Control who can read and write
  • Control what is data management
  • Have indexing support (for example B-Tree and R-Tree)

RDBMS Types

There are three RDBMS types in the world

  1. Embedded
  2. "Desktop"
  3. Server

Embedded databases

  • Part of an application
  • Have small footprint, and fast actions
  • Provide simple means to work with the data (Simple SQL Dialect)
  • Have limited data storage
  • In memory and/or disk storage.

"Desktop"

  • Can be used by several applications, but limited to a single access at a time.
  • Stored inside of the application(s) partition.
  • Have query language (SQL dialect).
  • Manages resources for both reading and writing records.
  • Disk storage.
  • A bigger resource footprint then "Embedded".

Server

  • Client Server based
  • Can have more then a single connection/usage
  • Have a query language (SQL dialect)
  • Some contain socket for events
  • Some contain replication support
  • Some contain robust setting mechanism for tweaking performance

SQLite

It's the answer to my question:
What is the most popular RDBMS in the world?

It's used on all mobile device today (Android and iOS).
It's used by Microsoft, Linux distro and MacOSX.
Even your web browser uses it internally.

Well-Known Users of SQLite

In short:
SQlite is a C library for embedded ACID* RDBMS.

* Atomic, Consistency, Isolation, Durability

The database is develop by D. Richard Hipp (working at General Dynamics) for a US-Navy contract in 2,000.

It was designed for guided missile destroyer software.

The database started as a TCL extension (a programming language).

The Syntax based on PostgreSQL 6.0, using B-Tree and transitions.
It support most of SQL92 standard.

Quirks and interesting SQLite facts

  • Version 3 is the current version.
  • It support Unicode functionality.
  • Weak typed - Field type is not enforced - string can be inside an int field.
  • Reading can be made using more then a single application at the same time, in parallel.
  • Writing can be made only by a single connection.
  • There is a WAL (Write-Ahead log) support.
  • It has support for full text search.
  • The file format can copied between 32 and 64 bit, and even big and little endian without a problem.
  • All of the design and structure for the database exists on SQLite web site.
SQLite web site:
https://www.sqlite.org/

Apache Derby

Apache Derby (used to be IBM Cloudscape), is both embedded and server RDBMS.

Original creator is Nat Wyatt from Cloudscape Inc. The first release named JBMS back in 1997.

In 1999 Informix Software Inc bought Cloudscape Inc. In 2001 IBM acquired the database from Informix Software.

The database was re-brand by IBM as "IBM Cloudscape" and released as embedded Java database, as part of IBM's Java stack.

Quirks and interesting Derby facts

  • Derby has small footprint (3.5 MB) for engine and JDBC driver.
  • Based on SQL standards.
  • Support JDBC driver connection.
  • Support Client/Server connection.
  • Lacks of Multi Version Concurrency (important for Client/Server ACID).
  • Sub project for Apache DB.
Derby web site:
https://db.apache.org/derby/

Firebird SQL

On 25 of July, 2000 the company Borland released their database engine - Interbase v6.0 as open source.
A week later, a new project was created named Firebird using Interbase v6.0 as their engine.

On 11 March 2002 Firebird v1.0 was released, as independent database, supporting multiple operating systems such as Linux, Windows, Mac and other Unix flavors, that were given support two months later.

A massive porting of the database was made from C to C++, and on 23 February 2004, Firebird 1.5 was released as a stable release.

v1.5 had better query optimizer, support for SQL-92 standard and some of SQL-1999 standards as well (transaction savepoint) and support for explicit locking.

On 12 November 2006 v2.0 was released, adding support for 64-bit architecture, procedural triggers, recursive queries and support for the MERGE (aka UPSERT) statement.

v2.5 introduced three main new features: Multithreading engine, REGEX syntax for query language and making query for a remote database.

Firebird 3.0 (the current engine) released on 19 April 2016, providing better security features, better performance and better SMP support.

Quirks and interesting Firebird SQL facts

  • Mature RDBMS architecture - The original database started on 1981.
  • Free to use and to change using IPL (InterBase Public License) and IDPL (Initial Developer's Public License).
  • Cross Platform RDBMS (Linux, Windows, FreeBSD, Mac and HP-UX) + Native 32 and 64 bit architecture.
  • Multi-generation architecture - reading does not lock writing.
  • Full ACID support.
  • "Read-Only" RDBMS support.
  • Supported as Embedded and Server.
  • Rich and powerful Query language support:
    • ANSI SQL support.
    • Support for CTE
    • Strong Transaction management system
    • Strong Procedural SQL support.
    • Cross-database queries
    • Triggers and Events
    • Domains
    • Views
    • User Defined Functions
    • Full cursor support
    • Clustered Indexes
    • Computed Fields
    • Sequence/Generator
  • Logging and Security features:
    • Real-time monitoring
    • Audit trails for events, partial and full logging
    • Support sending logging to remote server
    • SQL Debug
    • Support for Windows Trusted Authentication - Active Directory and SSO
    • User Roles, grants, access rights
    • Connection aliases - instead of addresses
  • Small footprint (between 4Mb to 33Mb depends on installation)
  • Backup and restore
  • Thread-safe client libraries
  • Supports up to 20 Tera bytes of data
  • Choice of architectures to fit all needs (without changing your application):
    • Embedded - A dll/so file.
    • SuperServer - single daemon/server for all client connections, multi threaded with shared cache
    • SuperClassic - single daemon/server for all client connections, multi threaded with separate caches
    • Classic - uses inetd to run one copy of the server per client connection, recommended for SMP systems but might have event-notification issues if access is via a firewall
  • And some less known stuff:
    • Not distributed
    • File stored locally to the server
    • Less common and less known
    • Used as embedded db for LibreOffice's Base program (Access like)
Website:
https://firebirdsql.org/

Usually we read it as MySQL/MariaDB, but let me start with history to explain the database, and why.

Using Wikipedia for that.

In 1994 MySQL was created by a Swedish company, MySQL AB, founded by David Axmark, Allan Larsson and Michael Widenius.

The first version of MySQL appeared on 23 May 1995.

It was initially created for personal usage from mSQL based on the low-level language ISAM, which the creators considered too slow and inflexible.

MySQL AB created a new SQL interface, while keeping the same API as mSQL.

MariaDB

On 2008 Sun Microsystems acquired MySQL AB.

On 2010 Oracle acquired Sun Microsystems.

The day that Oracle announced about the purchase of Sun Microsystems, Micael Widenius forked MySQL and launched MariaDB, and took most MySQL developers with him.

Today, Most Linux distro and so does open source Unixes (BSD) replaced MySQL with MariaDB!

The reason is more complicated then you think, and will not be addressed in this lecture.

MySQL quirks and interesting facts

  • Dual license - GPL (General Public License) and Commercial EULA.
  • Open source and Enterprise versions
  • A broad subset of ANSI SQL 99
  • Written in C and C++
  • Uses multi-layered server design with independent modules.
  • Designed to be fully multi-threaded using kernel threads, to easily use multiple CPUs if they are available.
  • Contains multiple types of storage engines.
  • Provides transactional and non-transactional storage engines.
  • Designed to make it relatively easy to add other storage engines. This is useful if you want to provide an SQL interface for an in-house database.
  • Uses fast B-tree disk tables (MyISAM) with index compression, but lack of transactions.
  • Uses a very fast thread-based memory allocation system.
  • Executes very fast joins using an optimized nested-loop join.
  • Implements in-memory hash tables, which are used as temporary tables.
  • Implements SQL functions using a highly optimized class library that should be as fast as possible. Usually there is no memory allocation at all after query initialization
  • Provides the server as a separate program for use in a client/server networked environment and as embedded/library environment.
  • Support Auto increment for numeric fields.
  • Does not support Sequences/Generators
  • Updatable views
  • Online DDL when using the InnoDB Storage Engine.
  • Transactions with savepoints when using the default InnoDB Storage Engine. The NDB Cluster Storage Engine also supports transactions.
  • ACID compliance when using InnoDB and NDB Cluster Storage Engines
  • Nested Select
  • SSL support
  • Query caching
  • Build-in replication engine
  • Full-text indexing and searching
  • Unicode support
  • Partitioned tables with pruning of partitions in optimizer
  • Shared-nothing clustering through MySQL Cluster
  • Commit grouping, gathering multiple transactions from multiple connections together to increase the number of commits per second.
  • Amazing documentation
  • Native JSON support (since v8.0)
  • Security:
    • ACL host-based verification.
    • Password security by encryption of all password traffic when you connect to a server.
    • MySQL Server supports a keyring service that enables internal server components and plugins to securely store sensitive information for later retrieval.

MariaDB quirks and interesting facts

  • Dual License - GPL and LGPL (Lesser General Public License)
  • Contains most features of MySQL.
  • Have support for Window functions.
  • Have Support for JSON field type (thanks to PostgreSQL developers).
  • Additional Storage engines that arrives by default.
  • Compatible with MySQL 5 series.
  • Support of external authentication systems.

PostgreSQL

In 1973, IBM started working on a database named System R, and released a series of white-papers that describes the system.

Two Berkeley scientist - Michael Stonebraker and Eugene Wong read the papers, and started working on a relational database research project named University INGRES.

In 1982, Michael Stonebraker left Berkeley to make a proprietary version for INGRES.

On 1985 Michael Stonebraker returned to Berkeley and started to work on a post-INGRES project to address issues found on the database design.

The project was named PostinGres. The new implementation try to "understand" relationship, and could retrieve information in related tables using rules.

Postgres used ideas from INGRES but did not used it's code.

On 1989 version 1 of Postgres was released, on 1990 version 2 and on 1991 version 3.
Version 2 and 3 rewrote how the rule system worked, and added support for multiple storage managers.

On 1994, after version 4 was done, Berkeley had release Postgres under an MIT like license, that helped other developers to use the code for any use case they desire.

Version 4 also supported POSTQUEL query language interpreter, and a cli tool named monitor to interact with the engine.

In 1994, Berkeley graduate students Andrew Yu and Jolly Chen replaced the POSTQUEL query language interpreter with one for the SQL query language, creating Postgres95.

The front-end program monitor was also replaced by psql. Yu and Chen announced the first version (0.01) to beta testers on May 5, 1995.

On July 8, 1996, Marc Fournier at Hub.org Networking Services provided the first non-university development server for the open-source development effort.

With the participation of Bruce Momjian and Vadim B. Mikheev, work began to stabilize the code inherited from Berkeley.

In 1996, the project was renamed to PostgreSQL to reflect its support for SQL.

The online presence at the website PostgreSQL.org began on October 22, 1996.

The first PostgreSQL release formed version 6.0 on January 29, 1997.

Quirks and interesting PostgreSQL facts

  • User-defined types
  • PostgreSQL License
  • It's ORDBMS - Object-relational database
  • Table inheritance
  • Foreign key referential integrity
  • Nested transactions (savepoints)
  • Multi-version concurrency control (MVCC)
  • Tablespaces
  • Point-in-time recovery
  • Basic Table Partitioning
  • Advanced support for Write ahead log (WAL)
  • Hot system configuration changing (ALTER SYSTEM)
  • Upsert support (Insert/update records)
  • Complex grouping functions (GROUPING SETS, CUBE, and ROLLUP)
  • JSON + JSONB field type support
  • Creating schema templates
  • Multiple procedural languages
  • Database transformation
  • Filtering aggregation functions
  • Autovacuum enabled by default
  • Changing column data types
  • Multiple Index types
  • Nested and recursive expression (CTE support)
  • Replication:
    • Logical Replication
    • Multiple synchronous standbys
    • Quorum commit for synchronous
    • Replication Slots
    • 3rd party replication engines
    • synchronize a PostgreSQL data directory with another data directory that was forked from the first one
  • Security:
    • Column level permissions
    • SCRAM-SHA-256 Authentication
    • SSL/TLS connection
    • Setting default permissions
    • GRANT/REVOKE ON ALL TABLES/SEQUENCES/FUNCTIONS
    • GSSAPI Authentication support
    • Large object access controls
    • Native LDAP authentication
    • Native RADIUS authentication
    • Per user/database connection limits
    • Roles

CUBRID

CUBRID (cube rid) is an open source (GPL server and BSD client) ORDBMS started on 2008 written by the company Naver Search Solutions.

It is built for the web world -> support vertical and horizontal scaling, high availability, caching mechanism, middleware and extensions.

The feature that distinguishes CUBRID database from other relational database systems is its 3-tier client-server architecture which consists of:

  1. Database server
  2. Connection broker
  3. Application layer

Quirks and interesting CUBRID facts

  • Database server:
    • Stores and manages data in multi-threaded client/server architecture
    • Processes the queries requested by users and manages objects in the database
    • Transaction management system
    • Action logging
    • Backup and restore operations
    • Data storage
    • Multiple database engines can be executed on the same machine
  • Connection Broker:
    • Relays the communication between the database server and external applications
    • Connection pooling
    • Connection monitoring
    • Log tracing
    • Analysis logs
    • Failover management
    • Caching results
    • Query syntax analysis including optimization and execution plan generation
  • Application layer:
    • API
    • Drivers for several programming languages
    • Tools such as Graphical user interface, command line tools and more
